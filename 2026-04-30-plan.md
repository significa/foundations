### 2. Radius scaling system
- Single base + derivations via `calc()` (Tailwind v4 `--spacing` pattern)
- **Decision pending: scale shape.** Options:
  - **Linear (2x):** `--radius * 1, 2, 4, 6, 8` — predictable, but perceptually uneven (gap between 6→8 feels smaller than 2→4)
  - **Fibonacci-ish:** `1, 2, 3, 5, 8, 13` — matches perception, each step ~1.6x prior
  - **Geometric (golden ratio ~1.618):** `1, 2, 3.2, 5.2, 8.4` — smoothest perceptually
  - **Tailwind-aligned:** `2, 4, 6, 8, 12, 16` — pragmatic, maps to existing utility names
- Research suggestion: render Button + Card + Modal + Dropdown side-by-side with each scale, pick visually. ~30min experiment before locking in.
- Once scale is picked: sweep components using literal `rounded-*`, add inner-radius math to Dropdown panel/items, Modal/inner surfaces, button-group adjacents.

### 3. `<Field.Control>` + form spacing + `Label` move
- Add `Field.Control` (Slot wrapper) reading `useField()`, injecting `id` + ARIA attrs into child
- **Move `Label` → `Field.Label`** (you're right, it doesn't earn a top-level export)
- **Solve form spacing in `<Field>` itself:** make `Field` a flex column with intrinsic `gap-N`. No more `<Label className="mt-2">`. Sub-components stay dumb.
- Final API:
  ```tsx
  <Field>
    <Field.Label>Email</Field.Label>
    <Field.Control><Input /></Field.Control>
    <Field.Description>...</Field.Description>
    <Field.Error />
  </Field>
  ```
- Delete `examples/input.tsx` (and siblings) — they were workarounds
- Open question: should `Field` accept a `gap` prop, or is one fixed value enough?

## Tier 3 — Pedagogic-aligned tooling

### CI validation of `page.mdx` dependencies
Idea: build-time script that fails CI if a component imports a sibling not declared in its `page.mdx`.

- Convention: declare deps in `page.mdx` frontmatter, e.g.:
  ```yaml
  dependencies:
    - components/slot
    - ui/popover
    - ui/divider
  ```
- Script:
  - Walk `src/foundations/<cat>/<name>/<name>.tsx`
  - Extract `@/foundations/...` imports
  - Compare to frontmatter `dependencies`
  - Fail if drift
- Add to `pnpm check` pipeline

Open questions:
- Frontmatter list vs parsed "Dependencies" section in MD body?
- Allowlist for blessed pairs (Dialog→Modal, Drawer→Modal, DatePicker→Calendar) — declare them and move on
- What about utility imports (`@/lib/utils/classnames`)? Probably skip — only check cross-component foundations imports.
